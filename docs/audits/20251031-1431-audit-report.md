### F-001 — Exposed Refresh Token Hashes in Git History

- **Severity:** Critical (CVSS 9.8)
- **CWE:** CWE-798 (Use of Hard-coded Credentials), CWE-312 (Cleartext Storage of Sensitive Information)
- **OWASP:** A02:2021 - Cryptographic Failures
- **Component:** `data/sessions.json`
- **Evidence:**
  - Tool: Gitleaks
  - Location: `/artifacts/audit/20251031-1431/gitleaks.json`
  - **24 secrets detected** in git history
  - All flagged as `generic-api-key` by Gitleaks
  - Examples:
    - `52e4c8d211422eaca0c6cab12a6c417415ca5c1466a1fd12ca835d499669b483`
    - `7d0eb32215992cfb4695f8f7d351114adade5290d24cf38e74ad7e098968d173`
  - Commits dating back to 2025-10-31
- **Impact:**
  - **Credential Exposure:** Refresh token hashes are accessible in git history, allowing attackers to reconstruct or brute-force original tokens
  - **Session Hijacking:** Attackers with access to token hashes can potentially forge valid refresh tokens
  - **Authentication Bypass:** Compromised refresh tokens grant persistent access to user accounts
  - **Lateral Movement:** Admin session tokens are exposed, enabling privilege escalation
- **Exploitation Scenario:**
  1. Attacker clones public/accessible repository
  2. Uses `git log` to access historical commits containing `data/sessions.json`
  3. Extracts refresh token hashes
  4. Attempts rainbow table or brute-force attacks to reverse SHA-256 hashes
  5. Uses compromised tokens to generate new access tokens
  6. Gains unauthorized access to user sessions

- **Fix Recommendation:**
  1. **IMMEDIATE:** Invalidate all existing sessions via `sessionManager.revokeAllSessions()`
  2. **IMMEDIATE:** Rotate all JWT secrets (`JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`)
  3. Remove sensitive files from git history:
     ```bash
     # Using BFG Repo-Cleaner
     bfg --delete-files sessions.json
     bfg --delete-files users.json
     git reflog expire --expire=now --all && git gc --prune=now --aggressive
     ```
  4. Update `.gitignore`:
     ```
     data/sessions.json
     data/users.json
     data/password_reset_tokens.json
     data/preferences.json
     *.log
     .env
     .env.local
     .env.*.local
     ```
  5. Implement secrets management:
     - Use **AWS Secrets Manager**, **Azure Key Vault**, or **HashiCorp Vault**
     - Store JWT secrets as environment variables only
     - Never commit secrets to version control
  6. Add pre-commit hook with Gitleaks:
     ```bash
     # .husky/pre-commit
     gitleaks protect --staged --verbose
     ```

- **Acceptance Criteria:**
  - ✅ All sessions invalidated and users re-authenticated
  - ✅ Sensitive files removed from entire git history
  - ✅ Gitleaks pre-commit hook passing with 0 secrets detected
  - ✅ Secrets stored in external vault (not in repository)
  - ✅ Updated `.gitignore` prevents future commits

### F-002 — Hard-coded Password Hashes in Version Control

- **Impact:**

  4. Runs offline cracking using tools like `hashcat` or `John the Ripper`
  5. Obtains plaintext passwords

  5. Encrypt database at rest:

  6. Add security headers:

### F-003 — Path Traversal Vulnerabilities

- **Severity:** High (CVSS 8.6)
- **CWE:** CWE-22 (Improper Limitation of a Pathname to a Restricted Directory)
- **OWASP:** A01:2021 - Broken Access Control
- **Component:** `client/src/lib/file-transport.ts`
- **Evidence:**
  - Tool: Semgrep
  - Rule: `javascript.lang.security.audit.path-traversal.path-join-resolve-traversal.path-join-resolve-traversal`
  - **15 instances detected** (WARNING severity)
  - Vulnerable lines: 154, 213, 220 (×2), 221 (×2), 232 (×2), 241 (×2), 296, 342, 382
  - User-controlled input passed directly to `path.join()` and `path.resolve()`
  - Examples:

    ```typescript
    // Line 154
    path.join(logDir, fileName); // fileName from user input

    // Line 220-221
    path.join(baseDir, `${level}-${date}.log`); // level influenced by user
    path.join(baseDir, `${level}.log`);
    ```

- **Impact:**
  - **Directory Traversal:** Attackers can read/write files outside intended log directories
  - **Information Disclosure:** Access to sensitive files (`.env`, `data/*.json`, source code)
  - **Arbitrary File Write:** Potential to overwrite critical application files
  - **Log Poisoning:** Inject malicious content into logs that may be executed later
  - **Privilege Escalation:** Overwrite config files to gain elevated access

- **Exploitation Scenario:**
  1. Attacker crafts malicious log level or component name: `../../../etc/passwd`
  2. Application constructs file path: `logs/../../../etc/passwd`
  3. `path.join()` resolves to `/etc/passwd`
  4. Attacker reads or overwrites system files
  5. Gains access to SSH keys, environment variables, or database credentials

- **Fix Recommendation:**
  1. Create path sanitization utility:

     ```typescript
     // lib/path-sanitizer.ts
     import path from "node:path";

     export function sanitizePath(basePath: string, userPath: string): string {
       // Remove null bytes
       const cleaned = userPath.replace(/\0/g, "");

       // Resolve absolute path
       const resolved = path.resolve(basePath, cleaned);

       // Ensure result is within base directory
       if (!resolved.startsWith(path.resolve(basePath) + path.sep)) {
         throw new Error("Path traversal detected");
       }

       return resolved;
     }

     export function sanitizeFilename(filename: string): string {
       // Remove directory traversal sequences
       return filename
         .replace(/\.\./g, "")
         .replace(/[/\\]/g, "")
         .replace(/[\x00-\x1f\x80-\x9f]/g, "")
         .trim();
     }
     ```

  2. Update `file-transport.ts` to use sanitization:
     ```typescript
     private getFileName(entry: LogEntry): string {
       const date = new Date().toISOString().split("T")[0];
       const baseDir = path.resolve(this.config.logDirectory);
       const sanitizedLevel = sanitizeFilename(LogLevel[entry.level].toLowerCase());

       const filename = this.config.rotateDaily
         ? `${sanitizedLevel}-${date}.log`
         : `${sanitizedLevel}.log`;

       return sanitizePath(baseDir, filename);
     }
     ```
  3. Implement allowlist for log file names:

     ```typescript
     const ALLOWED_LOG_LEVELS = ["debug", "info", "warn", "error", "fatal"];

     if (!ALLOWED_LOG_LEVELS.includes(sanitizedLevel)) {
       throw new Error("Invalid log level");
     }
     ```

  4. Add unit tests:
     ```typescript
     describe("sanitizePath", () => {
       it("prevents directory traversal with ../", () => {
         expect(() => sanitizePath("/logs", "../../../etc/passwd")).toThrow(
           "Path traversal detected",
         );
       });

       it("prevents absolute paths", () => {
         expect(() => sanitizePath("/logs", "/etc/passwd")).toThrow(
           "Path traversal detected",
         );
       });
     });
     ```

- **Acceptance Criteria:**
  - ✅ Path sanitization utility created and tested
  - ✅ All `path.join()` calls use sanitized inputs
  - ✅ Allowlist implemented for valid log levels
  - ✅ Unit tests pass with 100% coverage for sanitization functions
  - ✅ Semgrep scan returns 0 path traversal warnings

- **References:**
  - OWASP: A01:2021 - Broken Access Control
  - OWASP Path Traversal: https://owasp.org/www-community/attacks/Path_Traversal

### F-004 — Unsafe Format String Injection in Logging

- **Severity:** Medium (CVSS 6.1)
- **CWE:** CWE-134 (Use of Externally-Controlled Format String)
- **OWASP:** A01:2021 - Broken Access Control
- **Component:** Multiple logging modules
- **Evidence:**
  - Tool: Semgrep
  - Rule: `javascript.lang.security.audit.unsafe-formatstring.unsafe-formatstring`
  - **12 instances detected** (INFO severity)
  - Affected files:
    - `client/src/lib/error-logger.ts`: Line 91
    - `client/src/lib/file-transport.ts`: Lines 252, 398
    - `client/src/lib/logger.ts`: Lines 115, 118, 121, 125, 131, 176, 572, 577
    - `scripts/add-license-headers.js`: Line 48
  - Example vulnerable code:
    ```typescript
    // Line 91 in error-logger.ts
    console.error(`Error: ${userMessage}`, metadata);
    ```

- **Impact:**
  - **Log Injection:** Attackers can inject format specifiers to forge log entries
  - **Information Disclosure:** Log viewers may execute format strings, leaking memory addresses
  - **Log Forging:** Create fake log entries to hide malicious activity
  - **SIEM Evasion:** Poison logs to evade security monitoring and alerting
  - **Audit Trail Manipulation:** Compromise forensic investigations

- **Exploitation Scenario:**
  1. Attacker provides input: `%x %x %x %x %s` as error message
  2. Application logs: `console.error("Error: " + attackerInput)`
  3. Format specifiers may execute in log aggregation tools
  4. Attacker gains information about memory layout or stack contents
  5. Uses information for further exploitation (ROP chain, ASLR bypass)

- **Fix Recommendation:**
  1. Replace string concatenation with structured logging:

     ```typescript
     // Before (vulnerable)
     console.error(`Error: ${userMessage}`, metadata);

     // After (secure)
     logger.error("User error occurred", {
       userMessage,
       ...metadata,
     });
     ```

  2. Implement SafeLogger class:
     ```typescript
     class SafeLogger {
       private sanitize(input: any): string {
         if (typeof input !== "string") return String(input);
         // Remove format specifiers
         return input.replace(/%[sdifxXeEgGcponaA]/g, "");
       }

       error(message: string, metadata?: Record<string, unknown>): void {
         const sanitizedMessage = this.sanitize(message);
         const sanitizedMetadata = Object.fromEntries(
           Object.entries(metadata || {}).map(([k, v]) => [
             k,
             this.sanitize(v),
           ]),
         );
         logger.error(sanitizedMessage, sanitizedMetadata);
       }
     }
     ```
  3. Use structured logging throughout:

     ```typescript
     // Good practice
     logger.info("User login", { userId, timestamp, ip });

     // Avoid
     logger.info(`User ${userId} logged in from ${ip}`);
     ```

  4. Update all console.\* calls to use structured logger

- **Acceptance Criteria:**
  - ✅ All string concatenation in logging replaced with structured format
  - ✅ SafeLogger class implemented and integrated
  - ✅ Semgrep scan returns 0 format string warnings
  - ✅ Logging tests verify sanitization works correctly

- **References:**
  - CWE-134
  - OWASP: A01:2021 - Broken Access Control
  - Format String Attacks: https://owasp.org/www-community/attacks/Format_string_attack

### F-007 — Session Tokens Stored in Plain JSON Files

- **Severity:** High (CVSS 7.8)
- **CWE:** CWE-312 (Cleartext Storage of Sensitive Information)
- **OWASP:** A02:2021 - Cryptographic Failures
- **Component:** `server/auth/session-manager.ts`
- **Evidence:**
  - Sessions stored in `data/sessions.json`
  - Contains refresh token hashes, CSRF tokens, user agents, IP addresses
  - No encryption at rest
  - File permissions not restricted
  - Accessible to anyone with file system access

- **Impact:**
  - **Session Hijacking:** Attackers with file access can steal active sessions
  - **CSRF Token Bypass:** CSRF tokens stored in plaintext
  - **User Tracking:** IP addresses and user agents logged without consent (GDPR violation)
  - **Forensic Limitations:** No audit trail of session access

- **Fix Recommendation:**
  1. Migrate to database-backed sessions (addresses F-002 as well)
  2. Encrypt sensitive fields at application level:

     ```typescript
     import { createCipheriv, createDecipheriv, randomBytes } from "crypto";

     const ENCRYPTION_KEY = Buffer.from(env.ENCRYPTION_KEY, "hex"); // 32 bytes
     const IV_LENGTH = 16;

     export function encrypt(text: string): string {
       const iv = randomBytes(IV_LENGTH);
       const cipher = createCipheriv("aes-256-gcm", ENCRYPTION_KEY, iv);
       const encrypted = Buffer.concat([
         cipher.update(text, "utf8"),
         cipher.final(),
       ]);
       const authTag = cipher.getAuthTag();
       return `${iv.toString("hex")}:${encrypted.toString("hex")}:${authTag.toString("hex")}`;
     }

     export function decrypt(encrypted: string): string {
       const [ivHex, encHex, authTagHex] = encrypted.split(":");
       const iv = Buffer.from(ivHex, "hex");
       const encryptedData = Buffer.from(encHex, "hex");
       const authTag = Buffer.from(authTagHex, "hex");
       const decipher = createDecipheriv("aes-256-gcm", ENCRYPTION_KEY, iv);
       decipher.setAuthTag(authTag);
       return decipher.update(encryptedData) + decipher.final("utf8");
     }
     ```

  3. Store only hashed tokens:
     ```typescript
     // Never store plain tokens, only hashes
     const refreshTokenHash = hashToken(refreshToken);
     await db.insert(sessions).values({
       refreshTokenHash,
       // Other fields
     });
     ```
  4. Implement session storage best practices:
     - Use Redis for session storage (fast, secure, expires automatically)
     - Enable Redis authentication and TLS
     - Set proper TTLs on all session keys

- **Acceptance Criteria:**

  - ✅ Plain JSON files removed

  - ✅ File permissions audit passed

### F-009 — No Rate Limiting on Authentication Endpoints

- **Severity:** Medium (CVSS 5.9)
- **CWE:** CWE-307 (Improper Restriction of Excessive Authentication Attempts)
- **OWASP:** A07:2021 - Identification and Authentication Failures
- **Component:** `server/auth/jwt-auth-routes.ts`
- **Evidence:**
  - Manual code review
  - No rate limiting on `/api/auth/login`
  - No rate limiting on `/api/auth/register`
  - No account lockout mechanism

- **Impact:**
  - **Brute Force Attacks:** Unlimited login attempts enable credential stuffing
  - **Account Enumeration:** Attackers can discover valid usernames
  - **DoS:** Excessive requests overwhelm authentication system

- **Fix Recommendation:**
  1. Implement rate limiting (already installed: `express-rate-limit`):

     ```typescript
     import rateLimit from "express-rate-limit";

     const authLimiter = rateLimit({
       windowMs: 15 * 60 * 1000, // 15 minutes
       max: 5, // 5 requests per window
       message: "Too many authentication attempts, please try again later",
       standardHeaders: true,
       legacyHeaders: false,
     });

     app.post("/api/auth/login", authLimiter, loginHandler);
     app.post("/api/auth/register", authLimiter, registerHandler);
     ```

  2. Implement account lockout:

     ```typescript
     interface User {
       failedLoginAttempts: number;
       lockoutUntil?: Date;
     }

     // In login handler
     if (user.lockoutUntil && user.lockoutUntil > new Date()) {
       return res.status(423).json({ error: "Account temporarily locked" });
     }

     if (!validPassword) {
       user.failedLoginAttempts++;
       if (user.failedLoginAttempts >= 5) {
         user.lockoutUntil = new Date(Date.now() + 30 * 60 * 1000); // 30 min
       }
     }
     ```

  - ✅ Account lockout mechanism implemented

  - No CSRF middleware enforced globally

- **Impact:**
  - **Unauthorized Actions:** Attackers can forge requests on behalf of authenticated users
  - **Account Takeover:** Change email or password without user consent

     ```typescript
     const csrfProtection = csrf({

         sameSite: "strict",
       },
     });

     app.use(csrfProtection);
     ```

  4. Validate on server side for POST/PUT/DELETE requests
